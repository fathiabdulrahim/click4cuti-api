Tabbed shell concept

// src/components/profile/ProfileTabs.tsx
import { Tabs, TabsList, TabsTrigger, TabsContent } from "@/components/ui/tabs"
import { useSearchParams } from "react-router-dom"

export function ProfileTabs() {
  const [params, setParams] = useSearchParams()
  const tab = params.get("tab") ?? "personal"

  return (
    <Tabs value={tab} onValueChange={(v) => setParams({ tab: v })}>
      <TabsList>
        <TabsTrigger value="personal">Personal Detail</TabsTrigger>
        <TabsTrigger value="contact">Contact Info</TabsTrigger>
        <TabsTrigger value="dependents">Dependents</TabsTrigger>
        <TabsTrigger value="files">Files</TabsTrigger>
        <TabsTrigger value="training">Training</TabsTrigger>
        <TabsTrigger value="equipment">Assets</TabsTrigger>
        <TabsTrigger value="preferences">App Setting</TabsTrigger>
      </TabsList>
      <TabsContent value="personal"><PersonalDetailTab /></TabsContent>
      // ... etc
    </Tabs>
  )
}
Why URL-driven tabs: tabs as ?tab=files in the query string makes them shareable + the back button works. Avoid component-state-only tabs.

  marital_status: { SINGLE: "Single", MARRIED: "Married", DIVORCED: "Divorced", WIDOWED: "Widowed" },
  race:           { MALAY: "Malay", CHINESE: "Chinese", INDIAN: "Indian", OTHERS: "Others" },
  // ... etc
} as const
⚠️ API quirk to remember: Rails enum returns the lowercase key ("single") in JSON responses, but accepts the uppercase DB value ("SINGLE") on write. Your TypeScript types should reflect this on read vs write. Easiest: type as uppercase, normalize incoming with a small helper normalizeEnum(value): UPPER.

// src/api/admin/payroll.ts
export const payrollApi = {
  get: (userId: string) => api.get(`/admin/users/${userId}/payroll`),
  update: (userId: string, payload: PayrollUpdate) =>
    api.patch(`/admin/users/${userId}/payroll`, { payroll: payload }),

Testing

Project uses Vitest + React Testing Library. Per sprint, add at minimum:

  • One render test per new page (snapshot or critical-element check)
  • One mutation test per mutating hook (mock the API, verify invalidation key)
  • Form validation test for the largest new schema (Sprint 1 PersonalDetailTab)

Recommended kickoff: Sprint 0 (Foundation). Building ProfileTabs + AdminUserTabs + CrudListPage + FileUploadField + types first turns Sprints 3, 4, 8 into trivial drop-ins. After Sprint 0, sprints 1 and 2 are the next logical pair (they exercise the most novel UIs — tabbed profile, supervisor matrix, branches catalog). Sprints 3–5 can then be parallelized since they share the CrudListPage pattern.
