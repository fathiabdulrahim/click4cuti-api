Enum convention

Every closed-set string column uses a Rails enum. The DB check constraint stays as a defensive layer; the enum gives us querying ergonomics.

✅ Do this

class ClaimApplication < ApplicationRecord
  enum :status, {
    pending:   "PENDING",
    approved:  "APPROVED",
    rejected:  "REJECTED",
    cancelled: "CANCELLED"
  }
end

# Free automatic helpers:
ClaimApplication.pending           # scope
claim.pending?                     # predicate
claim.approve!                     # bang setter (updates DB)
ClaimApplication.statuses          # => { "pending" => "PENDING", ... }

❌ Don't do this (legacy pattern we've removed)

class ClaimApplication < ApplicationRecord
  STATUSES = %w[PENDING APPROVED REJECTED CANCELLED].freeze   # ❌ no scopes/predicates

  validates :status, inclusion: { in: STATUSES }                # ❌ duplicates DB constraint

  scope :pending,  -> { where(status: "PENDING") }              # ❌ enum gives this for free
  scope :approved, -> { where(status: "APPROVED") }

  def pending?  = status == "PENDING"                          # ❌ enum gives this for free
  def approved? = status == "APPROVED"
end

Prefix when the field name would collide

If an enum value would clash with another method (e.g. account_type: { current: ... } creates a method current? that clashes with the current SQL keyword pattern), use prefix::

enum :account_type, {
  saving:  "SAVING",
  current: "CURRENT",
  fixed:   "FIXED",
  others:  "OTHERS"
}, prefix: :account_type      # => account_type_saving?, account_type_current?, ...
