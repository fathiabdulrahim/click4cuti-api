3 Approval Hierarchy v2

Medium risk Leave-critical

Replace flat user_leave_approvers with multi-category, 2-level hierarchy. Refactor required in Leaves::ApplyService + Pundit policies.

⚠️ Data migration required: Copy existing user_leave_approvers rows into user_supervisors with category='LEAVE', level=1 before dropping old table.

New table user_supervisors

create_table :user_supervisors, id: :uuid do |t|
  t.references :user,       type: :uuid, null: false, foreign_key: true
  t.references :supervisor, type: :uuid, null: false, foreign_key: { to_table: :users }
  t.string  :category, null: false      # LEAVE, CLAIM, OVERTIME, TIMEOFF
  t.integer :level,    null: false      # 1 or 2
  t.timestamps
  t.index [:user_id, :category, :level], unique: true
  t.check_constraint "user_id <> supervisor_id"
end

Modify users— Yearly Form

add_reference :users, :ea_person_in_charge, type: :uuid,
              foreign_key: { to_table: :users }

Backfill script

UserLeaveApprover.find_each do |ula|
  UserSupervisor.create!(
    user_id: ula.user_id,
    supervisor_id: ula.approver_id,
    category: 'LEAVE',
    level: 1
  )
end
# Drop user_leave_approvers in a follow-up migration once service code switches over.

12 Payroll (Compensation)

High risk 3 sub-phases: 12a/12b/12c

Payroll data lives in three 1:1 association tables — keeps users lean and lets payroll endpoints/policies be scoped independently from core HR. Each model has belongs_to :user; User has has_one :bank_detail / :statutory_detail / :tax_relief.

Pattern: All three tables are 1:1 with users. Enforce uniqueness with unique: true on user_id. Use accepts_nested_attributes_for on User if you want to submit them in a single payroll payload.

12a. New table user_bank_details

create_table :user_bank_details, id: :uuid do |t|
  t.references :user, type: :uuid, null: false, foreign_key: true, index: { unique: true }
  t.string :bank_name           # enum: Maybank, CIMB, …
  t.string :account_number
  t.string :account_type        # SAVING, CURRENT, FIXED, OTHERS
  t.string :branch
  t.string :account_status      # ACTIVE, INACTIVE
  t.timestamps
end

12b. New table user_statutory_details

create_table :user_statutory_details, id: :uuid do |t|
  t.references :user, type: :uuid, null: false, foreign_key: true, index: { unique: true }

  # EPF
  t.string  :epf_number
  t.string  :epf_contribution_start       # BEFORE_1998_AUG, AFTER_1998_AUG, AFTER_2001_AUG

  # SOCSO
  t.string  :socso_number
  t.integer :socso_contribution_start_age

  # EIS
  t.decimal :eis_employee_rate, precision: 5, scale: 4
  t.decimal :eis_employer_rate, precision: 5, scale: 4

  # Income Tax
  t.string  :income_tax_number
  t.decimal :vola_amount, precision: 10, scale: 2

  t.timestamps
end

12c. New table user_tax_reliefs

create_table :user_tax_reliefs, id: :uuid do |t|
  t.references :user, type: :uuid, null: false, foreign_key: true, index: { unique: true }

  # Spouse
  t.boolean :spouse_is_working
  t.boolean :spouse_is_disabled
  t.string  :spouse_gender                # MALE, FEMALE

  # Relief flags
  t.boolean :contributes_to_sip, default: false, null: false
  t.string  :tax_category                 # REGULAR, REP, KNOWLEDGE_WORKER

  t.timestamps
end
Note: Children breakdown (under 18 / studying / disabled) is derived from family_members — not stored on user_tax_reliefs. Expose as model methods (children_under_18, children_studying, children_disabled) for PCB calc.

Model snippet

class User < ApplicationRecord
  has_one :bank_detail,      class_name: "UserBankDetail",      dependent: :destroy
  has_one :statutory_detail, class_name: "UserStatutoryDetail", dependent: :destroy
  has_one :tax_relief,       class_name: "UserTaxRelief",       dependent: :destroy

  accepts_nested_attributes_for :bank_detail, :statutory_detail, :tax_relief
end

Cross-cutting decisions to settle now

DecisionRecommendation
Split full_name → first_name + last_name?Yes; keep full_name as a model method (derived) until consumers migrate. Eventually drop the column.
Move bank/statutory/tax-relief into separate 1:1 tables?Yes — Phase 12 splits payroll data into user_bank_details, user_statutory_details, user_tax_reliefs. Keeps users lean (~28 new cols instead of 45), and lets payroll endpoints be scoped/audited independently.
user_leave_approvers migrationKeep table until Leaves::ApplyService switches to user_supervisors, then drop in a follow-up migration.
Pundit policiesEach new model gets a policy mirroring UserPolicy (tenant-scoped via company_id through user).
Audit (PaperTrail)has_paper_trail meta: { company_id: -> { user.company_id } } on every new user-scoped model.
Active StorageAlready configured. Use has_one_attached for single files (certifications, supporting docs), has_many_attached if multi-file later.
Multi-company accessDefer. Current users.company_id single-tenant model is fine for MVP. Add user_company_accesses join table only when first paying customer needs it.
Serializer viewsEach new model needs Default/Detail Blueprinter views. Admin endpoints use :admin view with audit data.
Seeds & factoriesUpdate db/seeds.rb + spec/factories/ for each new model. New required fields on users need factory defaults to avoid breaking existing tests.

Recommended start: Phase 1 (Job Info essentials + branches). Smallest, lowest-risk migration; immediately benefits leave logic by allowing probation employees to be flagged as ineligible for AL/SL until confirmed.
